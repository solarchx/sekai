- Multiple administrative levels (ADMIN, VP, TEACHER, STUDENT). *Admin = Superadmin, he who manage the DB and system workflow (L4... LAYER 4 REFERENCE???)

- Student, capable of viewing their schedules.... and grades? and they should also be capable of making a report during lessons.
>> Report is defined and considered as a form of presence within the subject. Hence, the rules of the report must deliberately state that the report must only be made by students present in said lessons.
>> Report should have another functionality. It is mandatory that the presence report should also include the 'presence rating' for the teachers themselves.
>> The teachers' presence % should be rated based on the averaged points they gathered during lessons for that semester.

- Teacher, capable of filling-in the grades of their students, viewing their own schedules, and their own presence rating.

- VP. Ah yes, the managers of the school. They should be capable of viewing practically everything, also given the capabilities to customize schedules, majors, subjects, classes, etc etc.
>> VP should be capable of adding new teachers alongside the previous lists of capabilities.

- Admin, also known as superadmin. Yeah, this one is basically the creator themselves... that's me and vendor here. We should have TOTAL ACCESS to the app and database.

Details for tables' structure:
All tables contain softDelete and timestamps from built-in Laravel.

- Users
>> Divided into 4 roles as stated above:
>>> STUDENT
>>> TEACHER
>>> VP
>>> ADMIN
A user table consist of roles (four of the above), identifier (their actual ID number (NIK or NISN)), the id of the class they're assigned to, and most basic Laravel table structure.

- Majors
>> The majors students can pick (basically what they wish to specialize in).

- Grades
>> The level of class a student is currently on.

- Classes
>> The class a student is assigned to. A grade may have multiple classes and a class usually depends a grade. A class also depends on the majors. It's some sort of join tables (class A on grade 12 on Science major).
>> A user, regardless of their rank can be assigned to a class. If it's a STUDENT, it will be treated as a class student and will automatically be assigned the activity (accounts for update and deletion tied to the class too) with the class id for that semester. Otherwise, the user will be treated as its homeroom teacher.

- Subjects
>> The subjects that can be taught within a class. It also depends on the major and grade the class is on. 
>> Due to the dependencies and its Many-To-Many relation (A subject may be taught in multiple majors, a major may teach multiple subjects), we should create two pivot tables to link subject-major and subject-grade.
>> However, if the classes of the same grade in the same major have variance in the array of subjects taught to the students of said classes. Then, the subject should only link to the classes itself via a pivot table (This improves flexibility, but also increase the tediousness in assigning each subject to each classes.)
>> Alternatively, we can keep both approach, making three pivot table (which is ridiculously hard to maintain its ACID properties) but only override the custom set of subjects in specific classes if the pivot relation exist in class-subject table.

- Semester/Academic Year
>> This is a table that denotes the year and semester for the academic time. It consists of academic year (eg. '2025/2026') and its semester (e.g 2).
>> This is a new factor that will definitely be used in lesson periods for even more consistency.
>> Reality shall not tear itself once more with the existence of time.

- Lesson/Activity Periods
>> The period for one lesson block. It has three dimensions: The beginning time, the ending time, the weekdays the period is located.

- Activities/Lessons
>> A 'class', the subject that takes place within a period for a class, taught by a specific teacher. Obviously, it depends on its period, class, teacher, and subject.

- Attendance Form
>> Teacher creates an attendance form everytime they enter the class. Attendance form serves as a form to hold all student attendance for that class period.
>> Attendance form should contain the date and activity of the class. It marks the existence of a study-teaching process of a class activity

- Student Attendance
>> This is where students can fill out their attendance. The attendance depends on the activity form and the students.

- Report/Presence for Activity
>> This is where the students can make report regarding the activity they're on. It contains the topic of the activity, its details, the score of the activity (0-3) depending on how present the teacher was.
>> This is linked to the student's attendance and must only be done after a student fill out their attendance.

- Score Distribution
>> This is a table to represent each lesson's different score weights and distributions.

- Student Grades
>> This is a table to store student grades information (No shit sherlock).

- Announcement
>> This is a table to store all the announcements.
>> An announcement can be made by a user with authority higher than STUDENT.
>> This table should contain the title of the announcement, the subtitle of the announcement, the content of the announcement, the sender of it, and the scope of the announcement.
>> Announcement scopes:
>>> SPECIFIC-CLASS: Announcement is broadcasted to all users tied to a specific class. (Via activity taught or homeroomed class)
>>> CLASS-TAUGHT: Announcement is broadcasted to all classes the user is tied to. (Via activity taught and homeroomed class)
>>> SPECIFIC-GRADE: Announcement is broadcasted to all users tied to a specific grade. (A user is tied to a specific grade based on the class they're assigned to)
>>> TEACHERS: Announcement is broadcasted to all users with TEACHER and above role. (Practically teacher to teacher)
>>> PUBLIC: Announcement is broadcasted to every user.

All tables in the database must be normalized, at least 3NF, preferably up to 5NF.
For detailed informations regarding the database structure, refer to tables.sql and the create_tables migration.

App Design and Functionality:
[Note]
Soft Delete:
- On Soft Delete, softDeletes the user (via the deleted_at field, Laravel should have a built-in handler for this).
- They may still be referenced on foreign keys relation and such, but may not be directly used or selected in the UX (Cannot logged in on an account that is softDeleted, cannot assign a user to a class that has been softDeleted, etc).
- In the admin pages, allows an option (a checkbox filter to hide or show) for the admins to display softDeleted rows and restore them.
- True delete should not be done for safety measures.
- Links/Relations of a softDeleted row that references it may be retained (in which they are also softDeleted, so it's a cascading softDelete).
- When recovered, the links/relations are also recovered.
- Links not retained will actually be deleted instead of softDeleted.
- softDeletes are usually executed via an action button on a table row of the read pages. It will prompt for confirmation, then softDeletes it.
- The button transforms into 'recover' button if the row has been softDeleted. When pressed, it recovers the row and its links.

Admins:
- Admins are defined as VP and ADMIN.
- VP stands for Vice-Principal and ADMIN stands for Superadmin.
- The real difference comes in when logging has been implemented (Only superadmin views logs).

Transaction:
- When a function has more than one step in the DB process, it is recommended to use transaction to retain ACID behavior.

READ PAGE:
- A page containing a table that displays datas of the subject (unless stated otherwise). With enough authority, each row will be armed with several action buttons.
- The columns would be the editable values column name (except for sensitive information like passwords).
- For example, a user READ PAGE would be a page containing a table with the columns of name, email, identifier, and role.
- On top of the table would be a total count of the rows and a button to perform the creation of a new row (via CREATE PAGE).

Pagination on Read Pages:
- Read pages in table format are paginated by 100.
- There must be pagination navigator buttons at the top and bottom of the table.

On Create/Read/Update/Delete:
- Keywords used to describe a resource management page or process of the subject (On Create describes the CREATE PAGE or process of the subject, etc).
